# Documentation That Transfers Knowledge

> Documentation exists to change the reader's brain state. If the reader leaves without understanding, deciding, or acting — the documentation failed.

---

## The Problem

Most documentation is written for storage, not transfer. Authors dump what they know into a page, mark it "documented," and move on. The result: walls of text that nobody reads, outdated pages that nobody trusts, and tribal knowledge that never escapes Slack threads.

The documentation was written. The knowledge was never transferred.

This guide exists to close that gap.

---

## Who This Is For

Anyone who writes documentation in any medium: README files, Confluence pages, wikis, internal portals, Notion docs, architecture documents, runbooks. The principles are universal — they work wherever humans consume written information under time pressure.

## The Single Test

> If a reader can act on your documentation after a 30-second scan — the documentation works.

If they can't, more words won't fix it. The problem is never "not enough information." The problem is always too much of the wrong information, or the right information buried under noise.
