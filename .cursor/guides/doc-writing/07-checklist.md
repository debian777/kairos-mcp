# Pre-Publish Checklist

Print this checklist. Check every box before you publish.

---

## Content

### Purpose and Scope
- [ ] Document has exactly one clear purpose
- [ ] Title clearly states what this is (5-10 words)
- [ ] I can explain in one sentence who this is for and what they'll learn

### Audience
- [ ] Primary audience identified and stated
- [ ] Prerequisites listed explicitly
- [ ] Expertise level appropriate for audience
- [ ] Jargon defined on first use (if audience is beginner)
- [ ] No assumed knowledge without stating it

### Structure
- [ ] Inverted pyramid: most important information first
- [ ] Summary (2-5 sentences) at the top
- [ ] Headers work as a standalone outline
- [ ] One main idea per paragraph
- [ ] Paragraphs are short (3-5 sentences)
- [ ] Progressive disclosure (summary → details → deep dive)

### Writing
- [ ] Active voice (not passive)
- [ ] Important information front-loaded in sentences
- [ ] Sentences are short (mostly under 25 words)
- [ ] Simple words ("use" not "utilize")
- [ ] Ban list phrases removed ("It is important to note that", "Please be advised", etc.)
- [ ] No condescending language ("obviously", "simply", "just")

---

## Visual Design

### Hierarchy
- [ ] Clear visual hierarchy (headers, emphasis, lists)
- [ ] Headers use proper levels (H2, H3, H4 - no skipped levels)
- [ ] Bold used for key phrases (not entire sentences)
- [ ] Code formatted with backticks or code blocks
- [ ] Code blocks have language tags

### Formatting
- [ ] Lists for multiple related items (not buried in paragraphs)
- [ ] Parallel structure in lists
- [ ] Tables for structured data, comparisons, or reference info
- [ ] Whitespace between sections
- [ ] Passes the squint test (not a wall of gray text)

### Visuals
- [ ] Diagrams for complex relationships (if needed)
- [ ] All images have alt text
- [ ] Screenshots are current (if using them)
- [ ] No unnecessary decorative images

---

## Navigation and Links

### Internal Links
- [ ] Links to prerequisites (what reader needs to know first)
- [ ] Links to related documents
- [ ] Links to detailed explanations (progressive disclosure)
- [ ] "Next steps" or "Related docs" section at the end
- [ ] All internal links work

### External Links
- [ ] External links are necessary (not gratuitous)
- [ ] External links open to stable URLs (not likely to break)
- [ ] Context provided for external links

---

## Metadata and Ownership

### Ownership
- [ ] Owner assigned (specific person, not team)
- [ ] Owner contact visible (name, email, or link to team)
- [ ] Last updated date visible
- [ ] Review frequency stated (monthly, quarterly, etc.)

### Metadata
- [ ] Date at top (not buried)
- [ ] Owner at top (not buried)
- [ ] Version number (if applicable)

---

## Accuracy and Maintenance

### Current State
- [ ] Information is accurate as of today
- [ ] Reflects current system state
- [ ] No outdated screenshots
- [ ] No references to deprecated features (unless marked as such)
- [ ] All commands and code examples tested and working

### Links
- [ ] All links checked (no 404s)
- [ ] No links to moved pages
- [ ] Relative links used for internal docs

### Future Maintenance
- [ ] Owner knows they're responsible
- [ ] Review date scheduled
- [ ] Document is findable (linked from index/parent)

---

## Accessibility

- [ ] Proper header hierarchy (no skipped levels)
- [ ] Alt text for all images
- [ ] Descriptive link text (not "click here")
- [ ] Tables have headers
- [ ] Sufficient contrast (if using colors)
- [ ] Not relying on color alone

---

## Platform-Specific

### Markdown (GitHub, GitLab, etc.)
- [ ] Fenced code blocks with language tags
- [ ] Tables formatted correctly
- [ ] Relative links for internal docs
- [ ] Images referenced correctly

### Confluence
- [ ] Used appropriate macros (Expand, Status, Panel)
- [ ] Table of contents macro (for long pages)
- [ ] Labels applied
- [ ] Page placed in correct space/hierarchy

### Wiki
- [ ] Follows wiki conventions
- [ ] Categories/tags applied
- [ ] Linked from relevant parent pages

---

## Final Review

### The 5-Second Test
- [ ] Someone unfamiliar can tell what this is about in 5 seconds

### The 30-Second Test
- [ ] Someone scanning can extract the key information in 30 seconds

### The Empathy Test
- [ ] Would someone from the target audience understand this?
- [ ] Would they know what to do next?
- [ ] Is there anything they'd skip? (If yes, delete it)

### The Squint Test
- [ ] Zoom out or squint - does it look like structured content or a wall of text?

### The Three-Pass Edit
- [ ] Pass 1 (Structure): Sections in right order, one idea per paragraph
- [ ] Pass 2 (Sentences): Active voice, front-loaded, short sentences
- [ ] Pass 3 (Words): Deleted 30-50% of original draft, removed filler

---

## Before You Hit Publish

- [ ] Had someone from target audience review it
- [ ] Tested any instructions (they actually work)
- [ ] Ran link checker
- [ ] Added to documentation index/navigation
- [ ] Notified team (if relevant)
- [ ] Set calendar reminder for first review

---

## Post-Publish

Within 1 week:
- [ ] Check reader feedback
- [ ] Fix any issues found
- [ ] Update if immediate corrections needed

Within 30 days:
- [ ] First review completed
- [ ] Ownership confirmed
- [ ] Review cadence working

---

**If you can't check all the boxes, either fix them or explicitly accept the debt.**

Don't publish broken documentation. Fix it, or wait until you can.
