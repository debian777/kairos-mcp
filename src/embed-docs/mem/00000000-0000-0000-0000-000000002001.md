# Create New KAIROS Protocol

## Activation Patterns

KAIROS protocol lifecycle — create, review, or fix adapters. Supports three operations:

- **create** — draft and mint a new adapter from scratch (offered when activate finds no match).
- **review** — audit existing adapters or adapter families for structural issues, missing cross-references, chaining gaps, DRY violations, or stale content. Read-only — produces a findings report.
- **fix** — apply specific corrections to an existing adapter: add missing layers, update cross-references, fix contract JSON, re-train. Mutates the adapter.

**Run this protocol when the user says ANY of:**

- "create a new KAIROS protocol" / "create new protocol adapter"
- "mint a workflow" / "create a new workflow"
- "review protocol" / "audit adapters" / "check protocol families for gaps"
- "fix protocol" / "update adapter" / "patch protocol"
- "create new protocol" (when activate found no match and user confirms)

**Trigger pattern:** **create** / **mint** / **review** / **audit** / **fix** / **update** / **patch** + (protocol / adapter / workflow) [ + (new / existing / KAIROS) ].

**Must Never:**
- Run when the user only asked to **execute** an existing protocol (use **activate** → **forward** (loop) → **reward**).
- Trigger when a strong match already exists from activate and the user wants to run it, not review or fix it.
- Mutate adapters during a `review` operation — produce findings only.

**Must Always:**
- Infer `{operation}` from trigger: create / review / fix.
- For `create`: confirm intent, then gather requirements, draft, review, and mint.
- For `review`: export the target adapter(s), audit against the checklist in Gather Requirements, produce a gap report. No `train` call.
- For `fix`: export the target adapter, identify the specific issue, draft the corrected markdown, get user approval, then `train` with `force_update: true`.
- Include Activation Patterns as the first H2 and Reward Signal as the last H2 in any newly drafted adapter.
- Apply DRY: no duplicated H2s, no boilerplate sections — fold shared content into steps or extract to a reference.
- Enforce 350-line max per file. If content exceeds this, split into linked adapters (via `mcp` challenge with `activate`) or extract a type extension / reference.

**Good trigger examples:**
- "Create a new KAIROS protocol for code review" → run this protocol (create)
- "No match found; I want to create a new protocol" → run this protocol (create)
- "Review all v4 protocol families for missing cross-references" → run this protocol (review)
- "Audit the gitlab bundle for chaining gaps" → run this protocol (review)
- "Fix the MR adapter — it's missing a branch validation step" → run this protocol (fix)

**Bad trigger examples:**
- "Run the standardize project protocol" → use activate/forward, not this
- "Search for deployment protocol" → use activate, not this
- "Create an MR" → use the MR adapter, not this

## Confirm Intent

Infer `{operation}` from the user's trigger:

| Signal | Operation |
|---|---|
| "create", "new", "mint", no match from activate | **create** |
| "review", "audit", "check", "gaps", "cross-references" | **review** |
| "fix", "update", "patch", "add missing", "correct" | **fix** |

For `create`: Ask "No existing protocol matched your query. Create a new one, or refine your search?"
For `review`: Ask "Which adapter(s) or bundle family to review?" (accept: adapter title, bundle directory name, or "all").
For `fix`: Ask "Which adapter to fix, and what's the issue?" (accept: adapter title or URI + description of the problem).


## Gather Requirements

### For `create` — collect details for a new protocol

**1. Determine the pattern:**

| Signal | Pattern |
|---|---|
| One workflow, no variants | **Standalone** — one protocol, all steps inline |
| Multiple issue types / output formats / media | **Router + Extensions** — router classifies, routes via `activate` |
| Domain has a decision tree (which type? which format?) | **Router + Extensions** |
| Reference material needed by steps | Extract to a separate file, not a protocol |

Real examples: Compose (writing router → 5 extensions), BNX/BIB (Jira routers → type-specific creation protocols).

**2. Collect per protocol:**

- **Title**: clear, descriptive (becomes H1)
- **Steps**: what the protocol does (each becomes H2 with a challenge)
- **Challenge type per step**: comment (agent output), user_input (human gate), mcp (tool call), shell (command)
- **Domain reference**: guides/docs the protocol should load?
- **Existing protocols**: `activate` for related adapters — reuse, compose, or extract shared content?
- **DRY check**: what content would be duplicated across extensions? Extract to a shared reference or the router body.
- **Size check**: will any single file exceed 350 lines? If yes, split into linked adapters or extract a reference now — do not plan to "trim later".

**3. If Router pattern:**

- What are the variants? (each becomes an extension protocol)
- What decision questions classify the variant?
- What types are forbidden?

### For `review` — audit existing adapters against this checklist

Export target adapter(s) via `export`. For each adapter, check:

1. **Cross-reference completeness** — does the adapter reference all related adapters it depends on? Look for implicit dependencies (e.g. an MR adapter that creates branches but never references the branch naming policy).
2. **Adapter handoff gaps** — if the adapter performs an action covered by another adapter (branch creation, Jira lookup, commit formatting), does it hand off to that adapter via `activate` or `forward`? Or does it silently skip the step, leaving the agent to improvise?
3. **DRY violations** — is policy content duplicated inline instead of linking to the canonical source?
4. **Contract coverage** — does every executable H2 have a contract JSON block? Are contract types appropriate (shell for observable commands, not comment)?
5. **Activation pattern accuracy** — do trigger phrases match what users actually say? Are Must Never/Must Always complete?
6. **Structure compliance** — first H2 is Activation Patterns, last H2 is Reward Signal, no duplicate H2 headings, under 350 lines.
7. **Sibling consistency** — for adapters in the same family (e.g. GitLab MR vs GitHub PR), are equivalent steps present in both?

### For `fix` — identify specific issue

Export the target adapter via `export`. Read the current markdown. Identify the exact gap or error. Plan the minimal change (add layer, update reference, fix contract JSON). The fix should not alter unrelated layers.

Summarize all gathered requirements or findings.

```json
{"contract":{"type":"comment","comment":{"min_length":50},"required":true}}
```

## Draft Markdown

### For `create` — draft new adapter markdown

Using the requirements, draft the full markdown for `train`.

### For `review` — draft findings report

Produce a structured gap report. For each finding: adapter title, gap type (from the review checklist), severity (high/medium/low), recommended fix (specific: which layer to add, which `activate` call to insert, which adapter reference to update). Save to `$KAIROS_WORK_DIR/protocol-review-findings.md`.

### For `fix` — draft corrected adapter markdown

Export the current adapter via `export`. Apply the identified fix to the markdown. Show a diff-style summary of what changed (layers added, references updated, contracts modified).

### Shared drafting rules (all operations)

**DRY — Don't Repeat Yourself:**

Every piece of knowledge must live in exactly one place in the protocol family. Before writing any content, ask: "Does this already exist in another protocol, a shared reference, or CLAUDE.md?"

- Boilerplate (auth handling, assignment workflow, error tables, validation checklists) belongs **inside the step that needs it** — one sentence, not a section. If it's identical across a protocol family, extract it to a shared reference file and load it via `activate(query: "...")` or a stored adapter URI.
- If two protocols share >50% of their body, extract the shared logic into a base protocol or reference.
- Every H2 must earn its place: if it has no challenge, it's not a step — fold its content into a step that does.
- Never duplicate an H2 heading within a file. KAIROS treats each H2 as a step — duplicates create phantom steps.

**350-line maximum per file:**

No protocol file may exceed 350 lines. Both humans and AI lose coherence on longer documents.

- If a protocol grows beyond 350 lines, **split it** — use an `mcp` challenge with `activate` or a stored adapter URI to link the continuation adapter, or extract domain-specific content into a reference/checklist file.
- Router protocols are naturally short (~50–80 lines). Extension protocols should target 100–200 lines.
- Type extensions and reference files (not full protocols) should stay under 150 lines.
- Measure after drafting: `wc -l <file>`. If over 350, refactor before minting.

**Generic-first — create generic, link from detailed:**

Create protocols from generic to detailed. The generic adapter comes first and must stand alone — it works even with zero extensions. Specializations (type checklists, domain extensions) are created second and link *back* to the generic via `activate(query: "...")` or a stored adapter URI.

- The generic protocol defines the workflow scaffold. The extension defines only what's different.
- Teaching is showing the difference: an extension that only contains the delta is easier to read, write, and maintain than a full copy with scattered overrides.
- Expansion is cheap — adding a new type means one new extension file. The generic adapters don't change. Zero regression risk on existing paths.

**Slug convention — deterministic protocol linking:**

Every protocol gets a `slug` — a short, unique, lowercase-hyphenated key for export, offline migration, and repo bookkeeping. Runtime linking uses `activate` or a stored adapter URI. The engine auto-generates the slug from the H1 title. Add explicit `slug:` frontmatter only when the auto-generated value is too long or ambiguous.

```yaml
---
slug: jira-markdown-formatting
---
# Jira Description Markdown Formatting via MCP
```

**When to use stored adapter URIs vs. activate:**

- `forward` with a stored `kairos://adapter/{uuid}` URI — deterministic adapter-to-adapter routing. Use when the author or runtime already knows the target URI.
- `activate` — user discovery and runtime type-dispatch when the target depends on the request text or only the adapter title is known at author time.

**Slug naming rules:**

1. **Families (router + extensions):** hierarchical. The router slug is the family prefix. Each child expands the parent by one hyphenated segment per routing level:
   - `jira` → `jira-bib` → `jira-bib-bug`
   - `compose` → `compose-email`
   - `standardize` → `standardize-analyze`
   - An agent can derive the parent slug by stripping the last segment. This enables mechanical traversal without search.

2. **Standalone protocols (no family):** action-oriented. Describe what the protocol does, not where it lives. The bundle directory is the namespace — the slug does not repeat it.
   - Good: `create-merge-request`, `makefile-setup`, `migrate-tf-to-git-sha`
   - Bad: `gitlab-create-merge-request` (redundant — it's already in `gitlab/`)

3. **Shared utilities:** prefix with the family they serve, at the family root level:
   - `jira-markdown-formatting` (used by all `jira-*` children)
   - `jira-alternative-mcp-unavailable`

4. **General rules:**
   - Lowercase, hyphen-separated. No underscores, no camelCase.
   - Short as possible while remaining unambiguous. Target 2–4 segments.
  - The slug is a durable bookkeeping key. Activation Patterns and adapter URIs handle runtime discoverability and routing.

**Protocol structure rules:**

- H1: protocol title (one H1 = one adapter)
- First H2: Activation Patterns
- Middle H2s: one per step, each ending with a ```json challenge block
- Last H2: Reward Signal

**Activation Patterns structure (required, exact order):**

1. Trigger phrases — 3–6 quoted phrases
2. Trigger pattern — verb + noun formula
3. Must Never — 1–3 cases (imperative, no "should")
4. Must Always — 1–3 mandatory behaviours
5. Good trigger examples — 2–3 with `→ run this protocol`
6. Bad trigger examples — 2–3 with `→ use X instead`

**Challenge types:**

| Type | Use for | Engine validates |
|---|---|---|
| `comment` | Agent extracts, analyzes, drafts, summarizes | min_length |
| `user_input` | Human confirms, approves, chooses | Non-empty response |
| `mcp` | Agent must call a specific tool | Tool called successfully |
| `shell` | Agent must run a command | Exit code 0 |

**Challenge JSON format** (place at end of each step). Use a **single** concrete `type` per step (`comment`, `user_input`, `mcp`, or `shell`). Do not use pipe-separated placeholders inside `type` — each ` ```json ` block with `{"contract":...}` becomes its own layer when minted.

```text
{
  "contract": {
    "type": "comment",
    "comment": { "min_length": 50 },
    "required": true
  }
}
```

**Router-specific steps:**

- Classify step: decision tree table, forbidden types, common misclassifications. Challenge: `user_input`.
- Route step: `activate(query: "<exact adapter title>")` per type, fallback field reference. Challenge: `mcp` with `tool_name: "activate"`.

**If Router pattern:** draft the router AND each extension as separate markdown files.

Post the full drafted markdown.

```json
{"contract":{"type":"comment","comment":{"min_length":100},"required":true}}
```

## User Review

For `create`: present the drafted protocol(s) to the user. Ask to approve, request changes, or cancel. If Router pattern, present all files (router + extensions).

For `review`: present the findings report. Ask user which findings to act on (fix now, defer, dismiss). If any findings are accepted for fix, switch `{operation}` to `fix` and proceed to Mint Protocol for each.

For `fix`: present the corrected adapter markdown with a summary of changes. Ask to approve the fix or request adjustments.

```json
{"contract":{"type":"user_input","user_input":{"prompt":"Approve the draft (create/fix) or acknowledge findings (review)"},"required":true}}
```

## Mint Protocol

For `create`: call `train` with the approved markdown. Use `force_update: false` unless the user explicitly asks to overwrite. If Router pattern: mint the router first, then each extension.

For `review`: no `train` call. Save the findings report to the repo under `reports/` if the user requests it. Protocol complete.

For `fix`: call `train` with the corrected markdown and `force_update: true`. Also update the corresponding repo file under `kairos-v4/bundles/` to keep the local copy in sync.

Report the outcome: adapter URI(s) for create/fix, or findings summary for review.

```json
{"contract":{"type":"mcp","mcp":{"tool_name":"train"},"required":true}}
```

## Reward Signal

Protocol complete when: `create` — adapter(s) minted and URI(s) reported. `review` — findings report delivered and acknowledged. `fix` — corrected adapter trained with `force_update: true` and repo file updated.
