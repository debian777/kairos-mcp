# Phase Critic

Bounded adversarial investigator fired as a subagent at phase boundaries. Receives a reference document, loads the calling protocol's invariants, and independently verifies the verification target by extracting claims, gathering evidence, and searching for contradictions.

## Natural Language Triggers

**Typically invoked by:** Any non-meta protocol at the end of Phase 1 (Plan) or Phase 2 (Implement) via `kairos_begin(key: "phase-critic")`.

**Can be invoked directly when agent needs:**

- "review phase output" / "verify plan" / "audit implementation"
- "adversarial review" / "critic review"

**Trigger pattern:** **review** / **verify** / **audit** / **critic** + (phase / plan / implementation / output).

**Must Never:**

- Modify any file, branch, or configuration outside `$KAIROS_WORK_DIR`. This is forensic validation, not execution.
- Trust the executor's assertions — verify independently from artifacts and current state.
- Classify a claim as `verified` based solely on failure to find contradiction. Positive supporting evidence is required.
- Accept "probably correct" or indirect inference as sufficient evidence for a `verified` verdict on high-risk claims.
- Rely on hidden prior reasoning state. Each run is stateless — reason only from artifacts and context available now.
- Continue deep verification after an early-abort trigger is detected.

**Must Always:**

- Extract claims under the defined extraction contract before investigating.
- Verify all in-scope claims before expanding investigation.
- Apply the evidence hierarchy — higher-priority evidence overrides lower.
- Search for contradictory evidence, not just confirmatory evidence.
- Include a confidence level (High / Medium / Low) in every verdict.
- Include a failure class in every FAIL verdict.
- Produce diagnostic artifacts on FAIL.

**Good trigger examples:**

- Implement protocol fires this after Phase 1 to validate the plan → run this protocol
- Compose protocol fires this after Phase 2 to verify the page was written correctly → run this protocol

**Bad trigger examples:**

- "Create a new protocol" → use `create-new-protocol`, not this
- "Implement a feature" → use `implement`, not this

## Artifact Contract

This protocol receives:

```
{
  reference_document: string (required) — path or content of the specification to verify against
  calling_protocol_slug: string (required) — slug of the calling protocol (to load Must Never / Must Always invariants)
  verification_target: string (required) — what to verify: "plan" | "codebase" | "output" | description of target
  context: {
    operation: string (optional) — e.g. "create" | "fix" | "review" | "update"
    work_dir: string (required) — $KAIROS_WORK_DIR path
    verdict_file: string (required) — filename for verdict output (e.g. "plan-review.md" or "impl-review.md")
  }
}
```

This protocol produces:

```
{
  verdict: "PASS" | "FAIL"
  confidence: "High" | "Medium" | "Low"
  failure_class: string (only on FAIL) — one of: plan_incomplete, implementation_incorrect, evidence_missing, contradiction_detected, external_dependency_failure, verification_blocked
  verdict_file: "$KAIROS_WORK_DIR/{verdict_file}" — structured audit report (agent mode) or conversational output (plan mode)
}
```

## Precondition Check [SUBAGENT]

Verify foundational preconditions before deep investigation. Early-abort if any are broken.

**Input:** `{reference_document}`, `{calling_protocol_slug}`, `{context}`.

**Actions:**

1. Verify the reference document exists and is readable. If missing or empty → early-abort.
2. Verify `$KAIROS_WORK_DIR` exists. If missing → early-abort.
3. Load the calling protocol's Must Never / Must Always sections via `{calling_protocol_slug}`. If protocol not found → warn but continue (invariants will be empty).
4. For Phase 2 reviews: verify the verification target is available (branch exists, files changed, output artifacts present). If the target state is fundamentally wrong (wrong branch, no changes when changes required) → early-abort.
5. Check that tools needed for verification are available (git, relevant MCP servers). If a required tool is unavailable and blocks verification → early-abort.

**On early-abort:**

- Set verdict to `FAIL`
- Set failure class to `verification_blocked`
- Set confidence to `Low`
- Write focused diagnostics explaining the precondition failure
- Skip all subsequent steps — do not attempt claim verification

**Output:** Preconditions verified, or early-abort with diagnostics.

```json
{"challenge":{"type":"comment","comment":{"min_length":50},"required":true}}
```

## Audit [SUBAGENT]

Perform bounded adversarial review of the verification target against the reference document and invariants.

**Input:** Reference document content, invariants (Must Never / Must Always), verification target access, `{context}`.

**Actions:**

**Step 1 — Extract claims under the extraction contract:**

From the reference document and invariants, extract every verifiable claim. Prioritise extraction as follows:

MUST extract and verify:
- Operational claims (steps, procedures, workflows)
- Actionable instructions (commands, API calls, tool invocations)
- Configuration and environment variable claims
- Architecture claims that imply behaviour
- Guarantees and constraints
- Deploy/runtime assumptions
- Security, auth, and permission claims

MAY ignore:
- Stylistic wording
- Non-operational descriptive framing
- Narrative text that carries no factual consequence

**Step 2 — Investigate with scope discipline:**

Required investigation order:
1. Verify all claims within the reference document scope first.
2. Fully review all high-risk claims inside scope with maximum depth.
3. Expand beyond scope only when:
   - Contradiction signals appear during in-scope review
   - A dependency outside scope is required to validate an in-scope claim
   - The claim involves a high-risk area

High-risk priority areas (always full-depth review):
- Setup steps and prerequisites
- CLI commands and flags
- Configuration keys and environment variables
- Authentication, identity, and permissions
- Deployment and release behaviour
- External publishing actions (Confluence, Jira, MR/PR)
- Data integrity assumptions
- Destructive or irreversible operations

**Step 3 — Gather evidence using the evidence hierarchy:**

For each claim, gather evidence from the highest-priority source available:

| Priority | Evidence source | Examples |
|----------|----------------|----------|
| 1 (highest) | Source code, config files, schemas, route definitions | Read actual files, trace imports, check types |
| 2 | Direct runtime verification | Shell command output, API response, generated artifact, git state |
| 3 | Tests | Test assertions, coverage reports, test output |
| 4 | MCP-queried system state | Confluence page content, Jira ticket fields, Context7 docs |
| 5 | External docs, web pages, linked references | URL validation, web fetch results |
| 6 (lowest) | Conversational assumptions or indirect inference | Not sufficient alone for `verified` |

Rules:
- Higher-priority evidence overrides lower-priority evidence on the same claim.
- Lower-priority evidence can support but not replace higher-priority evidence when higher-priority evidence is available.
- A claim verified only by evidence at priority 5-6 is `partially_verified` at best.

**Step 4 — For each claim, produce a verdict:**

```
CLAIM: <verifiable statement>
EVIDENCE: <exact proof — source, file path + line, command output, git ref, URL>
EVIDENCE PRIORITY: <1-6>
VERDICT: verified | partially_verified | unverifiable | incorrect
```

Verdict definitions:
- **verified** — positive supporting evidence exists at sufficient quality (priority 1-4) AND no contradictory evidence found.
- **partially_verified** — some evidence exists but incomplete, OR no contradiction found but positive proof is not strong enough, OR evidence is only from priority 5-6 sources.
- **unverifiable** — no sufficient evidence available from any source.
- **incorrect** — contradicted by evidence. Describe the contradiction.

Critical rule: A claim MUST NOT be classified as `verified` based solely on failure to find contradiction. Absence of counter-evidence without positive supporting evidence yields `partially_verified` or `unverifiable`.

**Step 5 — Search for contradictory evidence:**

For each claim, actively seek disconfirming evidence:
- Other call sites that depend on behaviour that was changed
- Constants, configs, or feature flags that override or interact with the claim
- Tests in other modules that implicitly rely on the old behaviour
- Git state inconsistencies (uncommitted files, wrong branch, merge conflicts)
- MCP/API responses that contradict stated outcomes
- Broken links or missing referenced resources

**Step 6 — Produce the overall verdict:**

Overall PASS requires:
- No claims with verdict `incorrect`
- No high-risk claims with verdict `unverifiable`
- Majority of in-scope claims at `verified` or `partially_verified`

Overall FAIL when:
- Any claim is `incorrect`
- Any high-risk claim is `unverifiable`
- Critical coverage gaps exist

**Output:** Per-claim audit results ready for verdict assembly.

```json
{"challenge":{"type":"comment","comment":{"min_length":100},"required":true}}
```

## Verdict Assembly [SUBAGENT]

Assemble the overall verdict, confidence level, failure classification, and diagnostic artifacts. Write to the verdict file.

**Input:** Per-claim audit results from the Audit step, `{context}`.

**Actions:**

1. Determine the overall verdict: `PASS` or `FAIL` (per Step 6 rules above).

2. Determine confidence level:
   - **High** — all high-risk claims verified with priority 1-2 evidence, comprehensive contradiction search completed, no unverifiable areas in critical scope.
   - **Medium** — most claims verified, some reliance on priority 3-4 evidence, minor unverifiable areas outside critical scope.
   - **Low** — significant reliance on weak evidence, multiple unverifiable claims, limited contradiction search depth, or high-risk areas with incomplete coverage.

3. On FAIL, classify the failure:

| Failure class | When to use |
|---------------|-------------|
| `plan_incomplete` | Plan does not cover all acceptance criteria or has gaps |
| `implementation_incorrect` | Code/output contradicts the plan |
| `evidence_missing` | Cannot verify critical claims — evidence unavailable |
| `contradiction_detected` | Positive evidence contradicts a claim |
| `external_dependency_failure` | MCP, API, or external service unavailable or returning errors |
| `verification_blocked` | Foundational precondition broken (from Precondition Check) |

4. On FAIL, produce diagnostic artifacts:
   - MCP verification failure → structured bug report to `reports/mcp-bug-<server>-<desc>-<date>.md` with: summary (one sentence: failure + server + tool), calls and responses in JSON code blocks (secrets redacted), reasoning (intent and interpretation), environment/context, reproduction steps.
   - Code/test issue → blocking issues with file paths + line numbers + root cause analysis in verdict file.
   - Plan gap → unverifiable claims, missing coverage areas, acceptance criteria not addressed in verdict file.
   - Dead links → list of broken URLs/references with expected vs actual status in verdict file.

5. Write the verdict file to `$KAIROS_WORK_DIR/{verdict_file}`:

For PASS:

```markdown
PASS
Confidence: High|Medium|Low

# Phase Critic: {verification_target}

## Claims Audit

### 1. <claim text>
- **Evidence:** <exact proof>
- **Evidence priority:** <1-6>
- **Verdict:** verified

### 2. <claim text>
- **Evidence:** <exact proof>
- **Evidence priority:** <1-6>
- **Verdict:** partially_verified
- **Gap:** <what is missing>

...

## Contradictions Searched
- <areas searched and findings — none or description>

## High-Risk Areas Reviewed
- <list of high-risk areas and their verdict>

## Confidence Rationale
- <why High|Medium|Low>

## Recommended Human Checks
- <things that cannot be proven from code or available tools>
```

For FAIL:

```markdown
FAIL
Confidence: High|Medium|Low
Failure class: <class>

# Phase Critic: {verification_target}

## Blocking Issues
1. <claim that failed>: <why, what evidence contradicts it>

## Retry Guidance
- <what to fix before retrying, informed by failure class>

## Claims Audit
...

## Diagnostic Artifacts
- <paths to any produced artifacts (bug reports, etc.)>
```

6. If in plan mode (native plan mode prevents file writes): present the verdict conversationally in chat instead of writing to file. Include all sections above.

**Output:** Verdict file written to `$KAIROS_WORK_DIR/{verdict_file}`, or conversational verdict in plan mode.

```json
{"challenge":{"type":"shell","shell":{"cmd":"test -f \"$KAIROS_WORK_DIR/$CRITIC_VERDICT_FILE\" && head -1 \"$KAIROS_WORK_DIR/$CRITIC_VERDICT_FILE\" | grep -qiE '^(PASS|FAIL)' && head -2 \"$KAIROS_WORK_DIR/$CRITIC_VERDICT_FILE\" | tail -1 | grep -qi 'confidence' && echo 'Verdict file valid'","timeout_seconds":5},"required":true}}
```

## Completion Rule

Only reachable after all prior steps are solved.

This protocol is complete when:
1. Preconditions checked — either verified or early-aborted with `verification_blocked`.
2. Claims extracted under the extraction contract from reference document and invariants.
3. Each claim investigated with scope discipline using the evidence hierarchy.
4. CLAIM → EVIDENCE → VERDICT produced for every extracted claim.
5. Contradictory evidence search performed for every claim.
6. Overall verdict (`PASS` or `FAIL`) determined with confidence level.
7. On FAIL: failure class assigned, diagnostic artifacts produced, retry guidance included.
8. Verdict file written (agent mode) or conversational verdict presented (plan mode).
