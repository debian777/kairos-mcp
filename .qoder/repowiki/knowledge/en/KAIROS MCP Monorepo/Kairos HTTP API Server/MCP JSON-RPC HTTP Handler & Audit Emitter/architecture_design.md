Three cooperating files form a thin HTTP-to-MCP bridge:
- `http-mcp-handler.ts` registers `POST /mcp`, performs bearer-token validation (when `AUTH_ENABLED`), enforces a process-wide concurrent request limit (`MAX_CONCURRENT_MCP_REQUESTS_RAW` resolved via `resolveMaxConcurrentRequests`), creates a fresh `StreamableHTTPServerTransport` + per-request MCP server for each call, runs it inside `runWithSpaceContextAsync` so tenant context is available, and wires `close`/`finish` lifecycle hooks for timeout/cancellation logging.
- `mcp-audit-emit.ts` is an isolated helper module exposing `installResponseCapture` (wraps `transport.send` to intercept JSON-RPC responses at the SDK level — not Express `res.json`), `generateCorrelationId`, and three emit functions (`emitRequestStart`, `emitRequestEnd`, `emitToolCallAudit`) that write structured audit lines via `writeAuditLine`. The capture is only installed when `AUDIT_LOG_LEVEL >= 3`.
- `mcp-ui-offerings-auth-jsonrpc.ts` provides implementation-defined JSON-RPC error envelopes in the -32000..-32099 range; `buildListOfferingsUnauthorizedJsonRpc` is used by the handler's special-case `listOfferingsForUI` method to return a machine-actionable auth-required response instead of letting the SDK respond with -32601 Method not found.

Dependency direction: handler → audit-emitter + ui-offerings-auth-jsonrpc → shared config/auth/build-version utilities. The handler owns transport lifecycle; audit helpers own event emission; the UI offerings file owns JSON-RPC envelope construction.
