Three cooperating files form a small pipeline around the forward/next flow:
- `next-proof-types.ts` declares shared DTOs (`ProofOfWorkSubmission`, `HandleProofOptions`, `HandleProofResult`, `ElicitResult`) consumed by both this module and the upstream `next-pow-helpers.js`.
- `next-previous-step.ts` provides two entry functions — `tryApplySolutionToPreviousStep` (applies a solution to the previous layer when the current layer has no stored inference contract) and `tryApplySolutionToPreviousStepWhenSolutionMatchesPrevious` (handles the PROOF_HASH_MISMATCH case where the client forwards with the next step's URI but actually answers the previous step). Both resolve the previous layer via `resolveAdapterPreviousLayer`, compute `expectedPreviousHash` (GENESIS_HASH for step 1 or the stored hash of the grandparent), and delegate validation to `handleProofSubmission`. `ensurePreviousProofCompleted` walks the chain back and returns a `PreviousProofBlock` with typed `error_code`s (`MISSING_PROOF`, `COMMAND_FAILED`) and a human-readable `next_action` tailored per proof type (`shell`, `user_input`, `mcp`, `comment`).
- `next-missing-proof-payload.ts` consumes that block and constructs the `MissingProofPayload` (`current_step`, `challenge` with the correct `proof_hash`, `next_action`, `retry_count`) so the client receives the previous step's nonce/challenge in the response body.

Dependency direction: this module depends on `next-pow-helpers.js` (validation/hashing), `adapter-navigation.js` (prev-layer resolution), `memory-accessors.js` (inference contract extraction), `proof-of-work-store.js` (nonce/hash/result persistence), `kairos-uri.js` (URI construction), and `memory-body.js`; it does not depend on HTTP handlers, keeping it reusable from both the forward bridge and the legacy HTTP route.
