Three cooperating files form a thin orchestration layer around `QdrantService`:
- `tune-execute.ts` is the entry point exporting `executeTune(qdrantService, input)`. It resolves adapter URIs to concrete layer UUIDs via `resolveAdapterIdFromWireUri`, groups multi-space layers by `space_id` through `selectAdapterLayerSet` (preferring caller-supplied or default write space), enforces protected-space guards (`assertWritableTuneLayers`), then either performs an in-place markdown-to-layers rewrite (`tuneAdapterMarkdownInPlace`, which parses frontmatter, runs `CodeBlockProcessor`, rebuilds via `buildHeaderMemoryAdapter`, validates H1-derived adapter ID and layer count, and persists per-layer `updateMemory`) or delegates to `executeUpdate` for patch-style updates. A final pass optionally reassigns all affected layers to a target space.
- `tune-messages.ts` provides two pure helpers — `rewriteTuneMessage` (capitalizes "memory" → "adapter layer" and normalises legacy `me|/m/` URI forms) and `buildTuneResultMessage` — so error messages from `executeUpdate` are rewritten into the new adapter-layer vocabulary before being returned.
- `tune-verify.ts` exports `verifyTuneLayerPersistence`, a post-write read-back check that re-fetches each updated layer from Qdrant and asserts its persisted `text` matches the expected value, guarding against silent cache no-ops.

Dependency direction: `tune-execute.ts` depends on both `tune-messages.ts` and `tune-verify.ts`; the latter two have no intra-module dependencies. All three depend only on shared services (`QdrantService`, `IDGenerator`, `CodeBlockProcessor`, `buildHeaderMemoryAdapter`, `parseFrontmatter`, `validateAdapterMarkdownSize`) and utilities (`kairos-uri`, `protected-space-write-guard`, `review-evidence-check`, `tenant-context`, `resolve-space-param`).
