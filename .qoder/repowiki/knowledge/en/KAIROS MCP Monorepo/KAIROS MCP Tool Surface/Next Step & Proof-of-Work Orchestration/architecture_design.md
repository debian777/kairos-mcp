Three cooperating files form the forward pipeline:
- `next.ts` is the entry point. It normalizes a KAIROS layer URI, loads the current Memory (with Redis-backed caching), attempts to apply the solution to a previous step via helpers in `next-previous-step.js`, then delegates validation/challenge construction to `next-pow-helpers.ts`. On success it resolves the next layer through `resolveAdapterNextLayer` and returns a `NextOutput` payload built by `buildNextPayload`; on failure it records quality metrics via `updateStepQuality`.
- `next-pow-helpers.ts` owns PoW semantics: `buildChallenge` (nonce rotation, TTL refresh), `handleProofSubmission` (type dispatch across `shell`/`mcp`/`user_input`/`comment`, nonce/proof_hash checks, idempotent result storage via `proofOfWorkStore`, retry counting up to `MAX_RETRIES`), and `tryUserInputElicitation` which calls back into the MCP server's `elicitInput` capability when the contract requires user confirmation. Error responses are produced by `buildErrorPayload`, which injects `next_action` guidance and local artifact directory fields.
- `shell-challenge-invocation.ts` is a pure, side-effect-free helper that translates a `ShellChallengeFields` record into a POSIX argv array, auto-injecting `-c`/`-e` for known interpreters (`python3`,`sh`,`perl`,`node`,`ruby`) and delegating to an `OTHER_STYLE` set (`terraform`,`docker`,`kubectl`, …) where the author supplies flags/cmd directly. It also provides `formatShellInvocationForDisplay` / `formatShellChallengeInvocationSummary` for human-readable output.

Dependency direction is one-way: `next.ts` → `next-pow-helpers.ts` → `next-proof-types.js` / `kairos-uri.js` / `local-artifact-dir-contract.js` / `embedding-service` / `proof-of-work-store`; `shell-challenge-invocation.ts` has no internal dependencies and is consumed only by display/UI code. The module never spawns processes — it builds the logical argv that executors run externally.
