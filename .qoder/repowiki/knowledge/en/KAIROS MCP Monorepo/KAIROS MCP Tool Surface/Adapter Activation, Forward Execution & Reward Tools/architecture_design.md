Three sibling MCP tool modules each expose a `registerXxxTool(server, ...)` entry point plus a pure `executeXxx(...)` function used by the server's tool dispatcher.

- `activate.ts` wraps `executeSearch`, canonicalizes adapter URIs via `buildAdapterUri`/`parseKairosUri`, resolves linked artifacts through `listAdapterArtifacts` + `mintExportArtifactDownloadCapability`, materializes them into `$KAIROS_LOCAL_ARTIFACT_DIR` with curl+sha256sum, and returns ranked choices with role-specific `next_action` hints. Refinement cycles are gated by `activateRefinementStore.incrementRefineCount(executionId)` so refine footers drop after two retries.
- `forward.ts` drives the layered execution graph: it loads a layer via `loadMemoryForParsedUri`, starts an execution in `forwardRuntimeStore` (and `executionTraceStore`) on first call, dispatches tensor vs. non-tensor contracts (`handleTensorForward`), elicits missing user input via `tryUserInputElicitation`, applies solutions to pending prior layers (`tryApplySolutionToPreviousStep`, `tryApplySolutionToPreviousStepWhenSolutionMatchesPrevious`), enforces chain integrity through `expectedPreviousHash` / `ensurePreviousProofCompleted`, and advances to the next layer via `resolveAdapterNextLayer`. Proof-of-work helpers live in `next-pow-helpers.js`; view builders in `forward-view.js`; trace persistence is fire-and-forget via `traceFireAndForget`.
- `reward.ts` evaluates the submitted outcome through `evaluateReward`, persists metrics via `applyRewardMetrics(qdrantService, ...)`, and writes structured reward records into `executionTraceStore.setReward`, wrapping storage failures as `REWARD_FAILED` KairosError(503).

Dependency direction is one-way: tools depend on services (`memory/store`, `qdrant/service`, `forward-runtime-store`, `execution-trace-store`, `proof-of-work-store`, `reward-evals`) but never on sibling tools except `activate` → `search`. All three share the same registration/metrics/error-shaping wrapper pattern around `server.registerTool`.
