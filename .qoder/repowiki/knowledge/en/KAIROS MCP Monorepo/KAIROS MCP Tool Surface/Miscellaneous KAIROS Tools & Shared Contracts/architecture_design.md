Three independent leaf files with no internal sub-packages:
- `delete.ts` exposes a pure `executeDelete(qdrantService, input)` function plus a `registerDeleteTool(server, name?)` adapter that wires it into the MCP server via `server.registerTool`, using `mcpLooseToolInput` + zod schemas (`deleteInputSchema`/`deleteOutputSchema`) for validation, `qdrantServiceSingleton` for persistence, and `mcp-metrics` / `tenant-context` for observability. The same `executeDelete` is also consumed by an HTTP route (per its JSDoc).
- `local-artifact-dir-contract.ts` is a zero-dependency contract module exporting the JSON field name `kairos_local_artifact_dir` and a builder that turns frozen config hints into a fresh mutable array so schema-typed responses can hold it.
- `review-evidence-check.ts` exports `assertReviewEvidencePassed(evidence)` which validates a `phase-critic` verdict against `TrainError` codes (`REVIEW_EVIDENCE_REQUIRED`, `REVIEW_GATE_FAILED`) and is reused by both `train` and `tune` to keep per-file line counts under ESLint's max-lines limit.
Dependency direction: these tools depend outward on `services/qdrant`, `utils/mcp-tool-doc-runtime`, `services/metrics`, `utils/tenant-context`, and `tools/train-store`; nothing in this scope imports back from them.
