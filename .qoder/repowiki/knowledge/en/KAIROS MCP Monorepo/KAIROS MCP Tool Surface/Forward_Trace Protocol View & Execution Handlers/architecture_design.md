Three cooperating files form the forward/trace protocol surface:
- `forward-view.ts` is the primary entry: it reads a `Memory`, normalizes its stored `InferenceContractDefinition`, builds a `ForwardOutput` (current layer, contract, next_call hint), resolves tensor inputs via `forwardRuntimeStore.requireTensorInputs`, and maps between internal `LayerPayload` and the public `ForwardOutput` shape. It also exposes helpers (`buildForwardUiSummary`, `extractUuid`, `mapProofSolution`) consumed by callers.
- `forward-trace.ts` handles the runtime side of a `forward` call: it validates a `ForwardSolution` against the layer's contract (especially `tensor` contracts with name/type/min/max checks), stores tensors in `forwardRuntimeStore`, computes a SHA-256 proof hash over the tensor payload, appends a trace record via `executionTraceStore.appendExecutionTrace`, then navigates to the next layer through `resolveAdapterNextLayer` and returns a new `ForwardOutput` via `buildForwardView`.
- `forward-helpers.ts` is a tiny shared utility module providing `buildEvidenceHint` and `buildEmptySolutionTemplate` for both view construction and solution scaffolding.

Dependency direction is one-way: `forward-trace.ts` → `forward-view.ts` → `forward-helpers.ts`. Cross-cutting concerns (memory store, Qdrant service, proof-of-work nonce reuse, local artifact dirs) are injected as parameters rather than imported directly, keeping the module testable and decoupled from concrete storage backends.
