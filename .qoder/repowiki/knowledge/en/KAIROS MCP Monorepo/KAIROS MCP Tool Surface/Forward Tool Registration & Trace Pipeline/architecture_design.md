Three cooperating files form the forward-tool surface:
- `forward-register.ts` is the entry point that calls `server.registerTool('forward', ...)` with schemas from `./forward_schema.js`, wiring Zod validation (`forwardInputSchema.safeParse`), OpenTelemetry-style metrics via `mcp-metrics.js` (calls, duration, input/output size, errors) keyed by tenant id, and error mapping to `formatForwardToolError`. On success it delegates to `executeForward` from `./forward.js`.
- `forward-tool-error.ts` is a thin serializer that flattens a `KairosError` into `{error, error_code, message, ...details}` consumed by the handler's catch branch.
- `forward-trace.ts` owns the persistence side of a forward pass: `appendExecutionTrace` writes an execution trace row through `executionTraceStore`; `handleTensorForward` validates a tensor against the layer's inference contract (type/length/item bounds, required inputs via `forwardRuntimeStore.requireTensorInputs`, condition evaluation), stores it in `forwardRuntimeStore`, computes a SHA-256 proof hash via `tensorProofHash` persisted in `proofOfWorkStore`, appends the trace, then resolves the next layer through `resolveAdapterNextLayer` and returns a view built by `buildForwardView`. A fire-and-forget helper wraps async persistence so failures only warn rather than fail the call. Dependency direction is one-way: register → executeForward → handleTensorForward → stores; no cross-imports between the three files.
