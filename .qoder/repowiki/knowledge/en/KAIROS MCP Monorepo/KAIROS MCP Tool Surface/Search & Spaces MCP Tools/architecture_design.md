Two sibling MCP tools share a common search pipeline:
- `search.ts` exposes `executeSearch`, the single entry point used by both the MCP tool and an HTTP route. It normalizes input (`queryForSearch` strips built-in protocol URIs/slugs), applies Redis-backed caching keyed on space + query + group-collapse flag + limit (TTL 300s via `redisCacheService`), then delegates to `doSearch`. `doSearch` runs a two-phase Qdrant search through `MemoryQdrantStore.searchMemories` (group-collapsed first pass, uncapped fallback if too few candidates), merges duplicates with head-layer/personal-space tie-breaking, sorts by score, and calls `generateUnifiedOutput` from `search_output.ts`.
- `search_output.ts` is a pure output formatter: `createResults` normalizes scores via `normalizePublicSearchScore` and filters by threshold; `resolveHead` resolves chain roots via `QdrantService`; `generateUnifiedOutput` builds the unified `{must_obey, message, next_action, choices[]}` shape, appending 'refine' and 'create' meta-footer choices whose protocol versions are resolved lazily from Qdrant in `resolveFooterProtocolVersions`.
- `spaces.ts` implements `executeSpaces` and `registerSpacesTool`. It scrolls each allowed space (from `getSpaceContextFromStorage().allowedSpaceIds` plus `KAIROS_APP_SPACE_ID`) via raw Qdrant `scroll`, groups points by adapter id, optionally enriches with titles/artifacts, and renders an optional HTML widget via `renderSpacesWidgetHtml`. The MCP registration wraps execution in `mcpLooseToolInput` validation, Prometheus metrics (`mcpToolCalls`, `mcpToolDuration`, `mcpToolErrors`, `mcpToolInputSize`, `mcpToolOutputSize`), and error normalization via `mcpToolInputValidationErrorResult`.

Dependency direction: both tools depend only on read-side services (`MemoryQdrantStore`, `QdrantService`, `adapter-navigation`, `memory-accessors`, `tenant-context`, `space-display`, `protocol-slug`, `kairos-uri`). They never mutate memory — they are read-only MCP tool handlers.
