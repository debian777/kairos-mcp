Three focused helpers that together implement one side of the export pipeline:
- `export-selection.ts` parses the raw `ExportInput` into a canonical list of `kairos://adapter/<id>` URIs, normalizing bare slugs, expanding `all_adapters` via `listAdapterUrisInSpace`, and enforcing `EXPORT_MAX_ADAPTERS`. It also provides pure helpers (`normalizeAdapterUri`, `dedupeSlug`, `rehomeItemFiles`) used by callers.
- `export-resolve-adapter.ts` resolves a single URI to `{ adapterId, layerId }`, preferring slug→UUID lookup through `QdrantService.findFirstStepMemoryUuidBySlug` when available and falling back to direct `MemoryQdrantStore.getMemory` calls. The optional `qdrantService` parameter makes this module work in both full-stack and memory-only modes.
- `export-source.ts` consumes the resolved adapter/layer/artifact URIs and returns an `ExportOutput`: artifact URIs are served directly from memory; adapter URIs return a JSON listing built via `listAdapterArtifacts`; any other URI is treated as a layer and its text content is returned.

Dependency direction is inward only — these files depend on `../services/memory/store.js`, `../services/qdrant/service.js`, `./kairos-uri.js`, `./artifact-catalog.js`, and `./skill-export/*`, but nothing outside this scope imports them except the export tool entry point. `executeExportSource` takes `resolveAdapter` as a function argument rather than importing it directly, decoupling the resolver from the source-fetcher so tests can inject a stub.
