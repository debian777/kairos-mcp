Three sibling tool modules each follow the same two-layer pattern: a thin `registerXxxTool(server, memoryStore, options)` function that mounts an MCP handler (input schema validation via Zod, tenant/space resolution, Prometheus-style metrics, and error normalization) and a pure `executeXxx(...)` function containing the business logic.

- `train.ts` — entry point for creating new adapters; resolves content from inline text or a `source_adapter_uri` fork via `QdrantService`, delegates persistence to `executeTrainStore` in `train-store.js`, and maps store results into output URIs through `resolveTrainOutputAdapterUri`. Errors are normalized by `formatTrainErrorPayload` into a stable `{error, message, details}` shape consumed by the caller.
- `tune.ts` — registers the `tune` tool; lazily imports `../services/qdrant/index.js` only when needed so the Qdrant dependency is optional at module load time. After `executeTune` writes, it calls `invalidateTuneCache` to keep downstream caches coherent.
- `update.ts` — provides a transport-agnostic `executeUpdate(qdrantService, input)` used by both the MCP tool and an HTTP route; validates each URI as a `layer` kind, enforces markdown size limits, extracts memory bodies, and returns per-row `{status, message}` with aggregate counters.

Dependency direction is one-way outward: these files depend on `../services/*`, `../utils/*`, and sibling `./train-store.js`, `./dump.js`, `./kairos-uri.js`, etc., but nothing inside those siblings depends back on them. Each file re-exports its `execute*` function so callers outside the MCP server can reuse the logic directly.
