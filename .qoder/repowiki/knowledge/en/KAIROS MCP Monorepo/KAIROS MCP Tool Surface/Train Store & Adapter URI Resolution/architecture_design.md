Three focused files form a thin orchestration layer between MCP/HTTP tool entry points and the persistence stack:
- `train-store.ts` exports `executeTrainStore`, the single train step that branches on MIME: when `normalizedMime` is non-null and not `text/markdown` it calls `memoryStore.storeArtifact(...)` with allowlist + size checks; otherwise it validates protocol structure via `validateProtocolStructure` and stores markdown adapters through `memoryStore.storeAdapter(...)`. A fork helper strips author `slug:` from frontmatter to avoid collisions. All failures are thrown as the domain-specific `TrainError(code, message, details?)`.
- `train-artifact-adapter-uri.ts` resolves an input `kairos://adapter/{slug}` to its canonical UUID-backed `kairos://adapter/{uuid}` by parsing/asserting the wire format and looking up the slug in `QdrantService.findFirstStepMemoryUuidBySlug`; missing service or slug raises `ADAPTER_LOOKUP_UNAVAILABLE` / `ADAPTER_NOT_FOUND`.
- `train-output-adapter-uri.ts` picks the output adapter URI from a precedence chain: explicit memory slug → parsed `itemAdapterUri` → parsed `inputAdapterUri` → normalized `memoryAdapterName` → raw `adapterId`, returning `undefined` when none apply.
Dependency direction is one-way: both URI helpers import `TrainError` from `train-store.ts`; neither imports the other. Both rely on shared URI builders (`buildAdapterUri`, `parseKairosUri`) from `./kairos-uri.js` rather than string concatenation.
