Tests are organized as one file per source unit under `tests/ui/`, split into two categories: component/integration tests that render with `@testing-library/react` (`App.test.tsx`, `HomePage.test.tsx`, `Layout.test.tsx`, `RunsPage.test.tsx`, `SolutionForm.test.tsx`, `RunGuidedContent.test.tsx`, `CopyButton.test.tsx`, `ErrorAlert.test.tsx`, `NotFoundPage.test.tsx`) and pure-function tests against UI utilities/hooks (`useActivate.test.ts`, `browse-adapters.test.ts`, `parseProtocolMarkdown.test.ts`, `rich-text-editor-markdown.test.ts`, `useThemePreference.test.tsx`). A shared `setup.ts` bootstraps the jsdom environment by polyfilling `localStorage` on both `globalThis` and `window`, mocking `navigator.clipboard`, extending Vitest's `expect` with `@testing-library/jest-dom` matchers, providing a global `React` import for JSX, and stubbing `react-i18next`'s `useTranslation`. `helpers.tsx` exposes `renderWithProviders`, which wraps rendered UI in `QueryClientProvider` (with retries disabled) and `MemoryRouter`; some tests still inline this setup. Tests import production code via `@/` path aliases from `src/ui/...`, keeping them co-located with the UI layer they exercise.
