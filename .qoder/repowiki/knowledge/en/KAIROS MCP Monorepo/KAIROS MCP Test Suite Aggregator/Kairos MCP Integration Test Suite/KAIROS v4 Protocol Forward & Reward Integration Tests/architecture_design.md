Each `.test.ts` file is an independent Jest `describe` suite that owns its own `beforeAll`/`afterAll` lifecycle to create and close an MCP client via `createMcpConnection()` from `../utils/mcp-client-utils.js`. Test fixtures are built on-the-fly by calling the server's `train` tool with Markdown protocols produced by `buildProofMarkdown(...)` from `../utils/proof-of-work.js`, then walking the protocol through successive `forward` calls until `next_action` points at `reward`. Shared assertions for activate choice shapes, slug validation, and deprecated-field checks live in `v4-activate-test-helpers.ts`; all parsed responses go through `parseMcpJson` / `withRawOnFail` from `../utils/expect-with-raw.js` so failures dump the raw call+result. Files under this scope cover: activation gating (`v4-kairos-activate-refinement-gating.test.ts`), first-call forward shape (`v4-kairos-forward-first-call.test.ts`), continued stepping (`v4-kairos-forward-continue.test.ts`), MCP contract args (`v4-kairos-forward-mcp-contract-args.test.ts`), v2 unified solution envelope (`v4-kairos-forward-v2-solution.test.ts`), reward result schema (`v4-kairos-reward.test.ts`), and generic activate helpers (`v4-kairos-activate.test.ts`). Dependency direction is one-way: test files → shared helpers → `tests/integration/utils/*`; no production source code is imported except for constants like `AUTHOR_SLUG_RE` and `MAX_PROTOCOL_SLUG_LENGTH` from `src/utils/protocol-slug.js` used to validate slug format.
