Two-layer structure:
- `harness/` defines a transport-agnostic `TestHarness` interface (`types.ts`) plus three concrete factories — `http-auth.ts`, `http-simple.ts`, `stdio-simple.ts` — each gated by runtime checks (e.g. `ENV=dev_stdio`, `AUTH_ENABLED=true`). A single `SCENARIOS` registry in `scenario.ts` enumerates supported configurations so assertion code reads capabilities from `harness.scenario` instead of raw env.
- Top-level `*.test.ts` files are per-feature suites (auth JSON-RPC envelope, client connection lifecycle, host/dynamic OAuth client groups, tools listing, UI resource read, HTTP concurrency/error/slug-forwarding). They either build their own `@modelcontextprotocol/sdk` clients directly or wrap a harness via `createMcpConnection()` from `../utils/mcp-client-utils.js`.
- `contracts/spaces-tool.contract.ts` is a transport-neutral contract function (`runSpacesToolContract`) parameterised by a `createHarness` factory; it is reused across scenarios to assert the same `spaces` tool shape and validation errors regardless of transport.
- Dependency direction: test files → harness factories → shared MCP client utils / Keycloak helpers; contracts depend only on the `TestHarness` interface, never on a specific transport.
