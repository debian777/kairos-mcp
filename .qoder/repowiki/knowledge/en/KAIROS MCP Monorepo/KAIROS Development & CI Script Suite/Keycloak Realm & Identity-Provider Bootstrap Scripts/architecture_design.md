Three independent entry points target a live Keycloak Admin API (no client library):
- `scripts/deploy-configure-keycloak-realms.py` is the primary orchestrator: it reads realm baselines from `scripts/keycloak/import/kairos-{dev,prod}-realm.json`, merges them into the current realm via GET+PUT to `/admin/realms/{realm}`, then performs post-merge fixes that realm-level PUT cannot reliably apply — specifically pushing `kairos-mcp` redirect URIs/web origins per port range, enforcing top-level groups plus the `shared/ci-test` hierarchy, creating the `openid` and `kairos-groups` client scopes with an OIDC Group Membership mapper (`full.path=true`), linking those scopes as default/optional on named clients, configuring trusted hosts and dynamic-registration allow lists, and finalizing test users so password grants succeed.
- `scripts/deploy-configure-keycloak-google-idp.py` manages the Google identity provider alias `google` under the configured realm by listing existing IdPs and issuing POST/PUT to `/admin/realms/{realm}/identity-provider/instances`; realms are deliberately excluded from realm JSON because realm PUT would overwrite them.
- `scripts/deploy-add-keycloak-demo-user.sh` is a thin bash wrapper that locates the Keycloak container via `docker compose ps -q keycloak` and runs `kcadm.sh set-password` inside it to create/update a demo user.

Dependency direction is one-way: all three scripts depend only on the running Keycloak Admin REST API; they do not import each other. Environment is loaded from `.env` at repo root plus `os.environ`, with defaults for `KEYCLOAK_URL`, `KEYCLOAK_REALM`, `TEST_USERNAME`, etc. The `scripts/keycloak/argocd/README.md` and `scripts/keycloak/import/README.md` document the intended usage but contain no executable code.
