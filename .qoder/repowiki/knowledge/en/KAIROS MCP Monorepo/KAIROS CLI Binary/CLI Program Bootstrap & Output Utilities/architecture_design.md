Three small, single-responsibility files that sit at the top of the CLI layer:
- `program.ts` is the entry point: it constructs a `Commander.Command`, attaches root-level flags (`--url`, `--timeout`, `--retries`, `--no-browser`) whose defaults come from `config.ts`, registers every subcommand via its `activateCommand(program)` / `forwardCommand(program)` … factory functions in sibling `./commands/*`, and uses a `preAction` hook to propagate merged globals into `process.env['KAIROS_API_URL']` / `KAIROS_NO_BROWSER` before each action runs. It also re-exports `createClientFromProgram` and `resolveClientOptions` from `client-factory.js` so callers can build an HTTP client from the parsed program.
- `config.ts` centralises the API base URL resolution order — explicit `--url` flag (already consumed by Commander), then `KAIROS_API_URL` env var, then persisted config via `config-file.js`, finally `http://localhost:3000` — and exposes both `getCliApiUrlDefault()` (used as the Commander option default) and a public `getApiUrl()` alias.
- `output.ts` is a thin wrapper around `process.stdout`/`stderr` written to bypass ESLint's `console.*` rule; it normalises error messages with a leading `Error:` prefix and offers JSON/markdown formatters for command output.

Dependency direction is one-way: `program.ts` → `config.ts` → `config-file.js`; `output.ts` has no internal dependencies. All three are imported only by the CLI bootstrap (`src/cli/index.ts` or equivalent) and by individual command handlers — they do not import business logic.
