Two parallel capability providers share an identical shape — one for skill-bundle zip downloads (`export-download-capability.*`) and one for single-artifact downloads (`export-artifact-download-capability.*`). Each provider is split into:
- A `*-capability.ts` file exposing `mint*Capability` / `verify*Capability` plus a `buildExport*DownloadUrl` helper. Minting creates a random id + expiry payload, base64url-encodes it, appends an HMAC-SHA256 signature (keyed by `KAIROS_EXPORT_DOWNLOAD_SECRET`), and persists the full record in the store with TTL.
- A `*-capability-store.ts` file defining the record interface and a thin `put(id, ttl)` / `get(id)` wrapper around the shared `keyValueStore` from `./key-value-store-factory.js`, using a module-scoped singleton instance.

Verification reverses the mint flow: splits on `.`, re-computes the HMAC, checks timing-safe equality, decodes the payload, validates the epoch expiry, then looks up the record by id and double-checks both id and server-side expiry. The token itself carries only `{id, exp}`; all downloadable metadata lives in the in-memory store keyed by `mem:export-*:` prefixes. Both flows inject `space_context` via `getSpaceContextFromStorage()` so downstream handlers can enforce tenant scoping. URL construction delegates to `resolvePublicExportBaseUrl()` from `../config/export-download-settings.js` and defaults to `/export/skill-zip/<opaque>` or `/export/artifact/<opaque>`.
