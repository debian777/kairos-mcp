Two cooperating sub-packages sit on top of the same Qdrant instance:
- `qdrant/` (`QdrantService`, `QdrantConnection`) is the thin, resilient HTTP client that owns connection lifecycle, collection aliasing, snapshots, quality metrics, protocol-step indexing and reward propagation.
- `memory/` (`store_adapters` + `memory_store_core`) builds higher-level concepts — parsing Markdown adapters into layered `Memory` records, embedding dense/title/activation-pattern vectors plus BM25 sparse fields, and exposing hybrid search with Redis-cached reads.

The two sides are wired together at construction time in `src/services/memory/store.ts`: `MemoryQdrantStore` creates its own `QdrantClient` (resolving the `current` alias via `resolveCollectionAlias`), then composes `MemoryQdrantStoreMethods` (core CRUD/search) and `MemoryQdrantStoreAdapter` (adapter/artifact ingestion). The adapter layer calls back into the methods layer to upsert points, while both layers share the same underlying client/collection so alias resolution stays consistent. Cross-cutting concerns (structured logging, config-driven URL/collection/apiKey, `CodeBlockProcessor`) are imported from shared `utils/` and `config.js` rather than duplicated.
