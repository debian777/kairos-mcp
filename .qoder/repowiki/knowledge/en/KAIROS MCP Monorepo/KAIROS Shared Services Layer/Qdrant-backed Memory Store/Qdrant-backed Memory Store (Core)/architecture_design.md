Four-file leaf module centered on the `MemoryQdrantStore` class in `store.ts`, which owns a `QdrantClient`, resolves an alias to a real collection name via `resolveCollectionAlias`, and delegates responsibilities to three collaborators:
- `store-methods.ts` (`MemoryQdrantStoreMethods`) — CRUD and search: point retrieval with space-scoped access control, hybrid query using Qdrant's prefetch+RRF over primary/title/activation dense vectors plus two BM25 legs, with a dense-only fallback; results are cached through `redisCacheService`.
- `store-init.ts` (`initializeQdrantStore`) — one-shot bootstrapping: creates the collection if missing, detects older single-vector layouts and migrates to named vectors (add → embed → remove), ensures BM25 sparse-vector config (with recreation-based fallback when `updateCollection(sparse_vectors)` is unsupported), creates full-text payload indexes, and backfills activation-search vectors.
- `memory-accessors.ts` — pure functions that normalize optional `adapter` metadata into stable fields (`getAdapterId`, `getLayerIndex`, `getChainRoot`, …) used across callers.

Dependency direction is strictly inward: `store.ts` → methods/init/accessors; methods depend on embedding service, BM25 tokenizer, tenant/space filter utilities, and the shared cache; init depends on `qdrant-utils` helpers and the embedding service. The `store.ts` constructor also wires a `MemoryQdrantStoreAdapter` (from `store-adapter.js`, outside scope) behind `storeAdapter`/`storeArtifact` entry points, keeping the adapter pipeline separate from the raw Qdrant methods.
