Two-stage pipeline centered on `store-adapter.ts`'s `MemoryQdrantStoreAdapter` class:

- Entry: `storeAdapter()` normalizes docs via `normalizeMarkdownBlob`, then tries header-based parsing (`buildHeaderMemoryAdapter` in `adapter-builder.ts`) which splits by H1/H2 sections and fenced contract blocks; if that yields memories it delegates to `storeHeaderBasedAdapter`; otherwise it falls back to one-memory-per-document via `storeDefaultAdapter`.
- Builder layer (`adapter-builder.ts`): `buildHeaderMemoryAdapter` + `processH1Section` split content by fenced contract blocks, extract Activation Patterns / Reward Signal sections, derive layer labels from H2 headings, and attach `inference_contract` metadata per segment.
- Persistence handlers (`store-adapter-default-handler.ts`, `store-adapter-header-handler.ts`): both follow the same upsert flow — slug allocation via `allocateAdapterSlugForMint`, duplicate-adapter guard via `handleDuplicateAdapter`, three-way embedding batch (primary / title / activation-pattern) through `embeddingService.generateBatchEmbeddings`, BM25 tokenization via `bm25-tokenizer`, Qdrant `upsert` with a fallback retry that drops the `bm25` vector when the collection lacks sparse config, Redis cache invalidation, and model-stats contribution recording.
- Validation helpers: `validate-protocol-structure.ts` enforces required H1, first H2 = "Activation Patterns", last H2 = "Reward Signal", at least one ```json contract block, no mixed fences, and allowed contract types; `validate-adapter-markdown-size.ts` caps document size before processing.
- Read-side converter: `qdrant-point-to-memory.ts` maps raw Qdrant points back to the in-memory `Memory` type with safe defaults.
- Shared plumbing: `store-adapter-helpers.ts` provides slug allocation, similarity checks, task/type derivation; `store-methods.ts` is injected for cross-cutting store operations; tenant/space context comes from `tenant-context.js`. Dependency direction is strictly inward toward `utils/`, `services/embedding/`, `services/stats/`, and `services/id-generator/`; nothing in this module is imported by sibling services except the adapter entry point.
