Single-file leaf module organized by concern rather than layer:
- `service.ts` exposes the public `EmbeddingService` class (`generateEmbedding`, `generateBatchEmbeddings`, `calculateCosineSimilarity`, `generateMemoryEmbedding`, `healthCheck`) plus a singleton `embeddingService` and the startup helper `probeEmbeddingDimension`. It orchestrates provider dispatch, metrics, tenant/request context, and audit.
- `providers.ts` implements HTTP transport for two backends — `postEmbeddingsOpenAI` (OpenAI `/v1/embeddings`) and `postEmbeddingsTEI` (Text Embeddings Inference) — behind a single `postEmbeddings(input)` dispatcher that honors `EMBEDDING_PROVIDER='openai'|'tei'|'auto'` and auto-falls back from OpenAI to TEI. All network calls go through `fetchWithRetries`, which applies exponential-backoff-with-full-jitter, a wall-clock retry budget (`EMBEDDING_RETRY_BUDGET_MS`), and respects `Retry-After` / `retry-after-ms` headers; non-retriable quota errors are surfaced as `EmbeddingRateLimitError`.
- `config.ts` derives endpoint URLs from top-level config and owns a process-wide `_resolvedDimension` cache set on first successful response and asserted on every subsequent call via `setResolvedEmbeddingDimension`/`getResolvedEmbeddingDimension`; callers must run `probeEmbeddingDimension()` at startup.
- `types.ts` defines the stable return shapes `EmbeddingResult` and `BatchEmbeddingResult` re-exported by `service.ts`.
- `audit.ts` centralizes structured logging of success/failure and anomaly detection (latency threshold, vector norm bounds, dimension mismatch) against thresholds from `../../config.js`, emitting both Prometheus counters (`../metrics/anomaly-metrics.js`) and structured log lines under `audit.embedding` / `audit.anomaly` categories.
- `bm25-tokenizer.ts` is a standalone pure function (`tokenizeToSparse`) producing FNV-1a hashed sparse vectors (30k buckets, sublinear TF values) for Qdrant sparse search — no dependency on the embedding providers.
- `health.ts` provides an independent health probe invoked via `EmbeddingService.healthCheck()`.

Dependency direction: `service.ts` → `providers.ts` → `config.ts`; `service.ts` also depends on `audit.ts`, `health.ts`, `../metrics/embedding-metrics.js`, and `../../utils/tenant-context.js`. Providers depend only on `config.ts` and shared logger/context utilities. No cross-imports between `bm25-tokenizer.ts` and the rest of the module.
